## Some context and roles:

* All `tall_canonical` rows reference `canonical_element.canonical_element_id`.
* Don’t invent branching “if this / if that” flows — keep the analysis linear. If a required detail is missing from, ask
  **one** precise clarifying question only.

* **I also confirm the following (don't assume otherwise):**
    * deterministic `canonical_element_id` rule present: ce.id is deterministic uuid generated from (templateUid,
      canonical_path, semantic_type, data_type).
    * repeat elements have thier ce.`data_type = Array`, and `semantic_type = Repeat`, and MultiSelect elements have
      their ce.`data_type`=Array, and `semantic_type=MultiSelectOption`
    * `semantic_type` can be one of: `OrgUnit`, `Team`, `Activity`, `Option` (Single), `Name`, `Age`, `PhoneNumber`,
      `Email`, `URL`, `Progress`, `Coordinate`, `Repeat`, `MultiSelectOption` (multi Option), `GeoJson`. (`OrgUnit`,
      `Team`, `Activity`, `Option` are system canonical entities and will be discussed in a later iterations, option
      value in a tall tables are mostly the `op.code`, but in old deprecated way it was `name_ar`, `name`, or `name_en`)
    * `data_type` can be one of: `TEXT`, `BOOLEAN`, `INTEGER`, `DECIMAL`, `TIMESTAMP`, `ARRAY` (`semantic_type.Repeat`,
      and `semantic_type.MultiSelectOption` are of data_type `Arrays`, and their values are also exploded in the tall
      table);
    * each `tc` row references a valid `ce`.
    * `tc.value_json` contains repeat children when repeat rows exist, and
    * deDup and idempotent upserts of tc rows: `On Conflict (instance_key, canonical_element_id)...`
    * `repeat_instance_id`, `parent_instance_id` (`repeat_instance_id` if inside a repeat, or `submission_uid` if at
      root submission, parent_instance_id = null is possible and considered at root level too), and `repeat_instance_id`
      is stable, globally unique repeat instance id, generated during first creation.
    * submission_uid, submission_id is always referencing the main submission uid, id (id is internal, uid is preferred
      for external).
    * contains all elements values, including the repeat instance as object "json" with the whole json of all repeats in
      the `value_json` column, as well as it's child elements values exploded each in its row with parent_instance_id
      set
      to the repeat_instance_id they belong to, notice: that multi select options in any level are also exploded with
      `repeat_instance_id`, and `parent_instance_id` roles staying the same and each value row is with deterministic
      `instance_key`.
    * all types of single value elements stored as text in `value_text`, as will as a typed value in the applicable type
      column. we can cast all data types from this cell or get it from the types col based on `ce.data_type`,
      in addition to the value is also stored typed in `value_bool`, `value_number`, `value_json`. this holds for all
      element.
